description: 변경된(또는 지정한) 자바 파일이 bcc 규약을 지키는지 검사(규약 준수 리뷰)
argument-hint: "[파일·디렉터리 경로 · GitHub PR URL — 생략 시 git 변경분]"
---

bcc가 지정한 **규약 준수 여부**를 판정하는 리뷰다. 목적은 일반 버그 헌팅이
아니라 "이 코드가 스킬(references)에 문서화된 규칙을 지키는가"의 확인이다.

1. **대상 결정**:
   - `$ARGUMENTS`가 **GitHub PR URL 또는 번호**면: `gh pr view <n> --repo <owner/repo> --json files -q '.files[].path'`로
     변경 파일을 수집해 `.java`만 남기고, `build/`·`generated`·`openapi` 등 **codegen 산출물은 제외**한다.
     파일 내용은 head SHA(`gh pr view <n> --json headRefOid`) 기준으로
     `gh api "repos/<owner/repo>/contents/<경로>?ref=<headSHA>" -q .content | base64 -d`로 받아
     **실제 줄번호로 인용**한다. 파일이 많으면 패키지 클러스터로 나눠 검토한다.
   - `$ARGUMENTS`가 **경로**면 해당 경로(파일·디렉터리)의 `*.java`.
   - 없으면 git 변경분 — `git diff --name-only HEAD -- '*.java'`에 `git status --porcelain`의 미추적 `*.java`를 더한다.
   대상이 없으면 그렇게 알리고 끝낸다.

2. **규칙 로드**: bcc:conventions 스킬(`bcc:conventions`)을 로드하고,
   대상 코드 성격에 맞는 references(도메인 로직 / 테스트 / Spring / 계층 / 네이밍 / 구조)를 읽는다.
   **계층·아키텍처 규약**은 프로젝트 아키텍처를 판별해 해당 스킬도 함께 로드한다 — 포트&어댑터면
   `bcc:hexagonal`, Uncle Bob 클린 아키텍처면 `bcc:clean`.
   **로드한 스킬(references·아키텍처)에 문서화된 규칙만이 지적의 근거**다.

3. **규약 준수 검토 (철칙)**:
   - **모든 지적은 references의 구체 규칙(`references/<파일>`의 해당 항목)을 반드시 인용**한다.
     특정 규칙에 못 붙이는 우려는 본문 findings에 넣지 않는다.
   - **단정 전 근거 확인.** 규칙 위반이라고 판정하기 전에 파일 내용·배선(`@Configuration`/데코레이터)·
     아키텍처 테스트 존재 등 판단 근거를 실제로 확인한다. 확인 못 한 것은 위반으로 단정하지 말고
     아래 "규약 외 참고"로 내린다 — 오탐 방지(예: 트랜잭션 경계는 서비스가 아니라 배선의
     `TransactionTemplate`에 있을 수 있고, 계층 경계는 접근제한자 대신 아키텍처 테스트로 강제될 수 있다).
   - **스켈레톤/인터페이스 우선 PR**에서는 "구현·검증 미완"을 위반으로 보지 않는다. 계약(반환
     타입·record 불변·네이밍·포트 도메인타입)만 검토한다.
   - 규칙 위반은 **버그 가능성 규칙**(금액 부동소수점, `null`/센티넬 반환, 가변 컬렉션 노출, 무효
     테스트, 경계 강제 없는 `public` 구현 등)을 **먼저**, 그다음 스타일 규칙 순으로.

4. **보고 형식**:
   - 파일별로 통과면 `PASS: <파일>` 한 줄.
   - 위반은 `[버그|스타일] <파일>:<줄> — <설명> (근거: references/<파일> · <항목>)`.
   - 규칙에 못 붙이지만 짚어둘 것(일반 버그 의심·보안·개인정보/전자금융 컴플라이언스 등)은 맨 끝
     **"## 규약 외 참고"** 섹션에 심각도 없이 간단히 적고, 본문 findings와 섞지 않는다.

5. **수정 금지**: 이 커맨드는 리뷰 전용이다. 코드를 고치지 말고, 수정은 사용자가 명시적으로
   요청할 때만 진행한다.
